# Plan: Add `hypt:docs` Skill

## Design Decisions

**What docs does:** Scans all project documentation and updates it to reflect the current branch/PR changes. A structured, proactive documentation pass that handles known doc types systematically.

**What docs does NOT do:**
- Update CHANGELOG with version entries (autoclose's job during version bump)
- Generate new documentation from scratch for undocumented features
- Touch AGENTS.md (auto-generated by sync script)
- Rewrite docs that are already correct
- Replace touchup's Step 5 doc pass (touchup is a lightweight reactive pass on ad-hoc docs; docs is a deeper structured pass — they complement each other)

**Scope — files the docs skill scans:**
- `docs/todos/backlog.md`, `TODOS.md`, `TODO.md`, `thoughts/todo.md` — checkbox completion
- `docs/roadmap.md` or similar — status/date updates, checkbox completion
- `README.md`, `plugin/README.md` — command/skill tables, feature lists
- `docs/**/*.md` — feature documentation affected by the PR
- Any `.md` file with `- [ ]` checkboxes — semantic matching against PR changes

**Context adaptation:**
- When called from pipeline (before PR merge): uses git diff against main + commit messages
- When called from close/autoclose (PR exists): uses PR title/body/files + commit messages
- Adapts automatically based on what's available

**Pipeline placement:** After review loop + unit tests (Step 4), before final save (Step 5). Docs reflect the final reviewed code and get included in the PR before it's ready to merge.

**Close/autoclose integration:** Replace Step 2 (inline checkbox logic) with a call to `hypt:docs`. Docs runs as Step 2 (check off completed + update docs), then suggestions runs as Step 3 (add new items) — correct ordering.

**Idempotency:** Each update is additive and re-checking produces no-ops (e.g., checking off an already-checked checkbox is harmless). If pipeline already updated docs, close's docs call simply finds nothing to change and skips the commit.

**Edge cases:**
- No documentation files exist: silently skip, don't fail
- PR touches code but no matching docs: skip those docs, don't generate new ones
- Project is brand new: skip all doc updates gracefully

**Commit strategy:** Single batched commit per run — `docs: update documentation for PR changes`. Skip commit if no files were modified.

**Routing phrases:** "update docs", "refresh docs", "check docs", "documentation", "scan docs"

- CHANGELOG is explicitly out of scope — autoclose handles version entries in Step 8.
